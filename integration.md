# Интеграция BAAZ CMMS ↔ BAAZ TMS (MVP)

Единый документ контракта интеграции. Объединяет два контура взаимодействия
между независимыми проектами Supabase — CMMS (АС ТОиР) и TMS (учёт инструмента).
Из объёма исключено всё, что помечено как post-MVP в исходных предложениях
(push-события выдачи, Realtime, batch-оптимизации через ETag) — они вынесены
в раздел «За рамками MVP» в конце документа.

---

## Резюме

| | |
|---|---|
| **Разделение** | Два независимых проекта Supabase. Общей БД нет. |
| **Контур А — Ремонт инструмента** | Инициирует **TMS**: отправляет инструмент в ТОиР. Воркфлоу заявки полностью в **CMMS**. |
| **Контур Б — Получение инструмента на наряд** | Инициирует **CMMS**: диспетчер резервирует инструмент под наряд. Заявка на выдачу полностью в **TMS**. |
| **Связь между контурами** | Независимы. Общее — только факт использования HTTP-интеграции между теми же двумя проектами. |
| **Транспорт** | HTTPS: Supabase Edge Functions (запись) + PostgREST/views (чтение). Кросс-базовых FK и триггеров нет. |
| **MVP-исключения** | Realtime, push-уведомления о выдаче/возврате инструмента (только polling), ETag-оптимизации — см. «За рамками MVP». |

---

## Принципы

1. **Каждая система — источник истины в своей зоне.** CMMS не ведёт каталог и остатки инструмента. TMS не дублирует статусы ремонтного воркфлоу.
2. **Именование не пересекается.** Идентификатор `tool_id` в контуре А — это **конкретный физический экземпляр** инструмента (мультиметр DT838, S/N SN-48291). В контуре Б используется **`catalog_item_id`** — идентификатор **типа/каталожной позиции**, а не экземпляра (серийный номер конкретного экземпляра присваивается кладовщиком позже, при резерве). Эти два идентификатора **не взаимозаменяемы** и указывают на разные сущности в БД TMS.
3. **Снимок вместо постоянной синхронизации.** Данные, переданные в момент создания заявки (название инструмента, номер наряда), фиксируются как снимок и не обновляются при последующем изменении исходной карточки.
4. **Идемпотентность создания — обязательна в обоих направлениях.** Повторный вызов с тем же идемпотентным ключом не создаёт дубликат.
5. **Интеграция только через HTTPS.** Аутентификация сервис-сервис — общий секрет (`Bearer`) + `apikey` Supabase, передаваемые в заголовках.
6. **Версионирование контракта.** Каждое тело запроса/ответа содержит `schema_version` (целое число, начиная с `1`) — задел на будущие несовместимые изменения контракта без необходимости менять URL.

## Общие технические требования

### Аутентификация

Заголовки на каждый запрос между системами:

```
Authorization: Bearer <INTEGRATION_SECRET>
apikey: <SUPABASE_ANON_OR_SERVICE_KEY>
```

Секрет свой для каждого направления (`CMMS_INTEGRATION_SECRET` на стороне CMMS,
`TMS_INTEGRATION_SECRET` на стороне TMS). Ротация секретов — вне рамок MVP,
согласуется отдельно при появлении такой необходимости.

### Идемпотентность

Все операции **создания** (создание заявки на ремонт, создание заявки на выдачу)
принимают обязательное поле `client_reference_id` (UUID), генерируемое
вызывающей стороной **до** первой попытки отправки.

**Важно:** если запрос формируется из нескольких вложенных идентификаторов
(например, `line_client_id` у строк заявки на выдачу), эти идентификаторы
должны генерироваться и сохраняться локально **один раз**, до первого вызова,
и переиспользоваться при повторной отправке из-за таймаута или сетевой ошибки.
Генерация новых `line_client_id` при каждой попытке ломает идемпотентность —
контракт полагается на стабильность этих значений между попытками.

Поведение при повторном вызове с уже известным `client_reference_id`: сторона,
принимающая запрос, возвращает `200 OK` с данными уже созданной записи —
дубликат не создаётся.

### Обработка ошибок и повторные попытки

Единая политика для всех запросов на создание/чтение между системами:

| Код | Причина | Действие вызывающей стороны |
|---|---|---|
| `200` / `201` | Успех | — |
| `400` | Ошибка валидации payload | Не повторять без исправления запроса |
| `401` / `403` | Неверный или истёкший секрет | Алерт разработчику, не повторять |
| `404` | Связанная запись не найдена (рассинхрон) | Не повторять; залогировать для ручного разбора |
| `409` | Конфликт (например, уже есть открытая заявка) | Не повторять; решение — по бизнес-правилу конкретного эндпоинта |
| `5xx` | Ошибка принимающей стороны | Повтор с экспоненциальным backoff, максимум 24 часа, затем dead-letter + алерт |

### Версионирование

```json
{
  "schema_version": 1,
  ...
}
```

Поле присутствует во всех телах запросов и ответов, описанных ниже
(в примерах JSON опущено для краткости, но обязательно к реализации).

# Контур А — Ремонт и поверка инструмента

Инициатор: **TMS** (инструмент отправляется в ремонт/поверку). Заявка живёт
и обрабатывается полностью в **CMMS** стандартным воркфлоу (`new → accepted →
in_progress → completed → closed`).

## Идентификация инструмента в заявке CMMS

Заявка CMMS (`public.requests`) описывает объект ремонта. Помимо привычных
`asset_id` (станок) и `location_description` (объект без записи в реестре),
появляется третий вариант:

| Объект | Поля CMMS |
|---|---|
| Оборудование (станок) | `asset_id` |
| Инфраструктура без записи в реестре | `location_description` |
| **Инструмент из TMS** | `tool_id` + снимок полей ниже |

### Поля снимка, которые TMS передаёт при создании заявки

| Поле | Обязательность | Назначение |
|---|---|---|
| `tool_id` | да | UUID **конкретного экземпляра** инструмента в TMS — постоянная связь |
| `tool_name` | да | Отображаемое название для очереди диспетчера |
| `tool_serial` | рекомендуется | Заводской/учётный серийный номер |
| `tool_type_name` | нет | Краткий тип («Мультиметр», «Калибр») |
| `title` | да | Причина заявки — что не так с инструментом. Не дублировать `tool_name` |
| `description` | нет | Дополнительные детали, обстоятельства |
| `request_type` | да | `inspection` или `service` |

Снимок фиксируется на момент создания и не обновляется при изменении карточки
инструмента в TMS.

**Не передаётся в CMMS:** склад, остатки, моточасы, полная карточка инструмента.

## REP-API-1 — Создать заявку на ремонт инструмента

**Метод:** `POST /functions/v1/integration-tms-create-request`
**Вызывает:** TMS

**Идемпотентность:** обязателен `client_reference_id` в теле (см. общие требования).

**Тело запроса:**

```json
{
  "client_reference_id": "f3a1b2c4-0000-0000-0000-000000000001",
  "tool_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "tool_name": "Мультиметр DT838",
  "tool_serial": "SN-48291",
  "tool_type_name": "Измерительный прибор",
  "request_type": "inspection",
  "title": "Истёк межповерочный интервал, показания дрейфуют",
  "description": "Последняя поверка 2024-06; при контрольном замере расхождение > допуска"
}
```

**Ответ (успех):**

```json
{
  "request_id": "…",
  "request_number": "З-2026-00142",
  "status": "new",
  "created_at": "2026-07-01T10:00:00Z"
}
```

**Поведение CMMS:**
- `INSERT` в `public.requests` с привязкой к `tool_id` и снимком полей;
- `status = new`, заявка появляется в очереди диспетчера;
- при повторном вызове с тем же `client_reference_id` — возврат существующей записи без дублирования.

**Валидация:** `title` — непустая строка после trim (`400` при нарушении).
CMMS не подставляет `title` из `tool_name` автоматически.

**Ошибки:** `409`, если для того же `tool_id` уже есть незакрытая заявка.

## REP-API-2 — Список заявок по инструменту

**Метод:** `GET /rest/v1/integration_tool_requests?tool_id=eq.<uuid>`
**Вызывает:** TMS (вкладка «История заявок ТОиР» на карточке инструмента)

**Поля ответа:**

| Поле | Описание |
|---|---|
| `request_id` | uuid |
| `request_number` | Номер для пользователя |
| `request_type` | `inspection` / `service` |
| `status` | Текущий статус |
| `title` | Причина заявки |
| `created_at` / `updated_at` | |

Сортировка: `created_at desc`.

## REP-EVT-1 — Webhook при закрытии, отклонении или отмене заявки на ремонт

**Инициатор:** Supabase CMMS (Database Webhook на таблице `requests`)
**Получатель:** Edge Function TMS

**Условие отправки:**
- `tool_id IS NOT NULL`;
- переход `status` в `closed`, `rejected` или `cancelled` — все три статуса обязательны к обработке.

**Тело payload:**

```json
{
  "event": "request.status_changed",
  "request_id": "…",
  "request_number": "З-2026-00142",
  "tool_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "previous_status": "completed",
  "new_status": "closed",
  "changed_at": "2026-07-01T15:30:00Z"
}
```

**Ожидание CMMS:** HTTP `2xx` от TMS. Повторы при ошибке — по единой политике
(см. «Общие технические требования»). TMS обязан быть идемпотентен по
паре (`request_id`, `new_status`) — повторная доставка того же события не
должна приводить к повторному разблокированию инструмента.

**Ожидаемая реакция TMS:** `closed`, `rejected` или `cancelled` → инструмент
снова доступен для выдачи (снимается пометка «в ремонте»).

# Контур Б — Получение инструмента на наряд

Инициатор: **CMMS** (диспетчер резервирует инструмент под наряд ремонта или
позицию графика ППР). Заявка на выдачу живёт полностью в **TMS**.

> Конвенция контракта: в JSON — machine-readable enum (`snake_case`, англ.).
> Подписи для UI («Готово к выдаче», «Выдано») — только на стороне
> приложений, не в самом API.

## Принципы контура

- Заявка на получение ≠ заявка на ремонт (контур А). Это независимые сущности в разных системах.
- **Резервирование, а не мгновенная выдача.** Диспетчер фиксирует потребность (`new`); кладовщик резервирует строки вручную.
- Один склад на заявку. Другой склад — отдельная заявка (повторный вызов ISS-API-1).
- Один наряд может иметь несколько заявок на выдачу (по одной на каждый нужный склад).
- Выдача и возврат — комплектом, всеми строками сразу.
- **Отмена — с любой стороны, всегда целиком по заявке.** Диспетчер CMMS — через `ISS-API-2` (batch). Кладовщик TMS — из своего UI (например, нет доступного инструмента, брак, списание); отдельного HTTP-вызова в CMMS **не требуется** — диспетчер узнаёт при опросе `ISS-API-5` (`status = cancelled`, `cancelled_by`, `cancel_reason`). Частичной отмены по строкам **нет**: если одна позиция недоступна, кладовщик отменяет **всю** заявку; диспетчер может создать новую заявку (другой склад, другой состав).
- CMMS не хранит остатки и серийные номера — только ссылку и краткий кэш статуса.

## Идентификация инструмента в заявке на выдачу

**`catalog_item_id`** — идентификатор **типа/каталожной позиции** инструмента
в номенклатуре склада TMS. Это не то же самое, что `tool_id` из контура А:
конкретный серийный экземпляр закрепляется кладовщиком уже после резерва
строки, на этапе `ISS-UC-2` ниже. Смешивать эти два идентификатора нельзя —
они ссылаются на разные таблицы в БД TMS (экземпляры vs каталог).

## Локальное хранение в CMMS

| Поле CMMS | Источник | Описание |
|---|---|---|
| `client_reference_id` | генерируется CMMS | Идемпотентность создания |
| `tms_requisition_id` | ответ TMS | UUID заявки в TMS |
| `warehouse_id` / `warehouse_name` | ответ TMS | Склад заявки, снимок названия |
| `cmms_request_id` / `cmms_schedule_id` | локально | Связь с нарядом |
| `last_known_status` | опрос ISS-API-5 | Кэш статуса для UI без обращения к TMS |
| `last_synced_at` | локально | Время последнего успешного опроса |
| `cancelled_by` / `cancel_reason` | опрос ISS-API-5 | При `cancelled` — кто отменил и почему (для баннера диспетчеру) |

**Не хранится в CMMS:** строки заявки, серийные номера, остатки склада.

## ISS-API-1 — Создать заявку на получение инструмента

**Метод:** `POST /functions/v1/integration-cmms-create-tool-requisition`
**Вызывает:** CMMS

**Идемпотентность (обязательно):**

| Механизм | Описание |
|---|---|
| `client_reference_id` (тело) | UUID, генерируется CMMS до первой попытки отправки, уникален в рамках интеграции |
| `Idempotency-Key` (заголовок) | Рекомендуется: `{work_order.kind}:{work_order.id}:{warehouse_id}:{client_reference_id}` |

При повторном вызове с тем же `client_reference_id` TMS возвращает `200`
с данными уже созданной заявки, без дубликата. `line_client_id` каждой
строки должен быть сгенерирован один раз и переиспользован при повторной
отправке — см. «Общие технические требования».

**Тело запроса:**

```json
{
  "client_reference_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "warehouse_id": "f1e2d3c4-b5a6-7890-abcd-ef1234567890",
  "work_order": {
    "kind": "request",
    "id": "b2c3d4e5-f6a7-8901-bcde-f12345678901",
    "number": "З-2026-00142",
    "status": "in_progress",
    "title": "Замена подшипника шпинделя",
    "asset_name": "Токарный станок 16К20",
    "location_name": "Цех №1, участок механообработки"
  },
  "technician": {
    "id": "c3d4e5f6-a7b8-9012-cdef-123456789012",
    "full_name": "Иванов Иван Иванович"
  },
  "lines": [
    {
      "line_client_id": "d1e2f3a4-b5c6-7890-abcd-ef1234567890",
      "kind": "catalog",
      "catalog_item_id": "d4e5f6a7-b8c9-0123-def0-234567890123",
      "quantity": 1
    },
    {
      "line_client_id": "e2f3a4b5-c6d7-8901-bcde-f12345678901",
      "kind": "free_text",
      "description": "Динамометрический ключ 1/2\" до 200 Н·м",
      "quantity": 1
    }
  ],
  "notes": "Срочно к началу смены"
}
```

**Правила payload:**
- `warehouse_name` не передаётся в запросе — только в ответах TMS;
- `kind=catalog`: обязателен `catalog_item_id`, поле `description` не передаётся (TMS подставит название из каталога);
- `kind=free_text`: обязателен `description`, `catalog_item_id` отсутствует (кладовщик привяжет номенклатуру при резерве);
- `line_client_id` — стабильный UUID строки от CMMS, используется для сопоставления после резерва;
- `quantity` — целое число экземпляров (инструмент штучный).

**Ответ (успех, всегда `new`):**

```json
{
  "requisition_id": "e5f6a7b8-c9d0-1234-ef01-345678901234",
  "client_reference_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "warehouse_id": "f1e2d3c4-b5a6-7890-abcd-ef1234567890",
  "warehouse_name": "Инструменталка РМУ",
  "status": "new",
  "created_at": "2026-07-01T08:15:00Z",
  "lines": [
    {
      "line_id": "f6a7b8c9-d0e1-2345-f012-456789012345",
      "line_client_id": "d1e2f3a4-b5c6-7890-abcd-ef1234567890",
      "line_status": "pending",
      "kind": "catalog",
      "catalog_item_id": "d4e5f6a7-b8c9-0123-def0-234567890123",
      "description": "Набор рожковых ключей 8–22"
    },
    {
      "line_id": "a7b8c9d0-e1f2-3456-0123-567890123456",
      "line_client_id": "e2f3a4b5-c6d7-8901-bcde-f12345678901",
      "line_status": "pending",
      "kind": "free_text",
      "description": "Динамометрический ключ 1/2\" до 200 Н·м"
    }
  ]
}
```

**Поведение TMS:**
- `INSERT` заявки, все строки `line_status = pending`, производный `status = new`;
- заявка попадает в очередь кладовщика выбранного склада;
- резерв — отдельное действие кладовщика (ISS-UC-2), автоматического резерва при создании заявки нет.

**Валидация:**
- `client_reference_id` — UUID, уникален;
- `warehouse_id` — существует и доступен интеграции;
- `lines[].catalog_item_id` (при `kind=catalog`) — существует в номенклатуре указанного `warehouse_id`;
- `work_order.kind` ∈ `{request, schedule}`, `work_order.id` обязателен;
- `work_order.status` ∈ `{in_progress, scheduled}`;
- `technician.id`, `technician.full_name` — обязательны;
- `lines` — непустой массив, у каждой строки — `line_client_id`, `kind`, и (`catalog_item_id` или `description`).

**Ошибки:**

| Код | Тело |
|---|---|
| `400` | `{ "error": "validation_failed", "details": [...] }` |
| `409` | Не используется на этапе создания — недостаток остатка проверяется только при резерве кладовщиком |

## ISS-API-2 — Отменить заявки на выдачу (batch, только CMMS)

**Метод:** `POST /functions/v1/integration-cmms-cancel-tool-requisitions`
**Вызывает:** **только CMMS** при отмене наряда или отдельных заявок диспетчером.

> Отмена **кладовщиком TMS** через этот endpoint **не выполняется** — кладовщик действует в UI TMS; CMMS читает результат через `ISS-API-5`.

**Тело (ровно один критерий):**

```json
{ "cmms_request_id": "b2c3d4e5-f6a7-8901-bcde-f12345678901", "reason": "work_order_cancelled" }
```

```json
{ "requisition_ids": ["e5f6a7b8-…", "a1b2c3d4-…"], "reason": "dispatcher_cancelled" }
```

**Допустимые значения `reason` (запрос CMMS):** `work_order_cancelled`, `dispatcher_cancelled`, `duplicate`.

**Ответ:**

```json
{
  "cancelled": [{ "requisition_id": "e5f6a7b8-…", "status": "cancelled" }],
  "skipped": [{ "requisition_id": "a1b2c3d4-…", "reason": "already_issued" }]
}
```

**Правила:**
- Отмена допустима, пока ни одна строка не перешла в `issued`/`returned` (производный статус ∈ `{new, partially_reserved, ready_for_issue}`);
- TMS фиксирует `cancelled_at`, `cancelled_by = dispatcher`, `cancel_reason` = значение `reason` из запроса, откатывает резерв затронутых строк;
- при отмене наряда CMMS делает один batch-вызов, не N отдельных запросов.

**Отмена кладовщиком (TMS UI, вне ISS-API-2):**

| Поле TMS | Значение |
|---|---|
| `cancelled_at` | момент отмены |
| `cancelled_by` | `storekeeper` |
| `cancel_reason` | `out_of_stock` \| `damaged` \| `decommissioned` \| `other` |
| `cancel_reason_text` | опционально; свободный текст (обязателен при `other`, рекомендуется в остальных случаях) |

Допустимо из тех же производных статусов, что и для диспетчера. После отмены заявка терминальна; CMMS прекращает опрос по ней.

## ISS-API-5 — Статус заявки на выдачу

**Метод:** `GET /functions/v1/integration-cmms-tool-requisition`

**Query-параметры:**

| Параметр | Описание |
|---|---|
| `requisition_id` | Одна заявка |
| `cmms_request_id` / `cmms_schedule_id` | Все заявки по наряду/позиции ППР |
| `fields` | `summary` (по умолчанию) \| `full` (строки и серийные номера) |

**Стратегия опроса CMMS:**
- на карточке наряда — один запрос `list-by-work-order` (`cmms_request_id=` или `cmms_schedule_id=`), не по каждой заявке отдельно;
- опрос выполняется только для нарядов с **активными** (не терминальными) заявками — для заявок в статусе `returned` или `cancelled` повторный опрос не выполняется;
- детали конкретной заявки (`fields=full`) — по клику пользователя.

**Ответ (`fields=summary`, фрагмент):**

```json
{
  "work_order": { "kind": "request", "id": "b2c3d4e5-…", "number": "З-2026-00142" },
  "requisitions": [
    {
      "requisition_id": "e5f6a7b8-c9d0-1234-ef01-345678901234",
      "warehouse_id": "f1e2d3c4-b5a6-7890-abcd-ef1234567890",
      "warehouse_name": "Инструменталка РМУ",
      "status": "issued",
      "issued_at": "2026-07-01T10:00:00Z",
      "lines_summary": { "total": 2, "pending": 0, "reserved": 0, "issued": 2, "returned": 0 }
    }
  ]
}
```

**Ответ (`fields=summary`, заявка `cancelled` кладовщиком, фрагмент):**

```json
{
  "requisitions": [
    {
      "requisition_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "warehouse_id": "f1e2d3c4-b5a6-7890-abcd-ef1234567890",
      "warehouse_name": "Инструменталка цех №2",
      "status": "cancelled",
      "cancelled_at": "2026-07-01T09:15:00Z",
      "cancelled_by": "storekeeper",
      "cancel_reason": "out_of_stock",
      "cancel_reason_text": "Динамометрический ключ 1/2\" — нет свободных экземпляров",
      "lines_summary": { "total": 2, "pending": 1, "reserved": 1, "issued": 0, "returned": 0 }
    }
  ]
}
```

### Производный `status` заявки

Вычисляется TMS при каждом чтении на основе `lines[].line_status` и `cancelled_at`:

| Условие | `status` |
|---|---|
| `cancelled_at` задан | `cancelled` |
| Все строки `pending` | `new` |
| Часть строк `reserved`, не все | `partially_reserved` |
| Все строки `reserved` | `ready_for_issue` |
| Все строки `issued` | `issued` |
| Все строки `returned` | `returned` |

### Статус строки `line_status`

| Значение | Смысл |
|---|---|
| `pending` | Ожидает резерва |
| `reserved` | Зарезервировано на складе |
| `issued` | У исполнителя |
| `returned` | Принято обратно на склад |

### Поля отмены (при `status = cancelled`)

Возвращаются в `requisitions[]` (и в ответе по одному `requisition_id`):

| Поле | Описание |
|---|---|
| `cancelled_at` | ISO 8601 |
| `cancelled_by` | `dispatcher` — через ISS-API-2; `storekeeper` — из UI TMS |
| `cancel_reason` | См. таблицу ниже |
| `cancel_reason_text` | Опционально; комментарий кладовщика или диспетчера |

| `cancel_reason` | Кто задаёт | Смысл |
|---|---|---|
| `work_order_cancelled` | dispatcher | Наряд отменён в CMMS |
| `dispatcher_cancelled` | dispatcher | Диспетчер снял заявку вручную |
| `duplicate` | dispatcher | Дублирующая заявка |
| `out_of_stock` | storekeeper | Нет доступного инструмента для резерва/выдачи |
| `damaged` | storekeeper | Инструмент непригоден (брак) |
| `decommissioned` | storekeeper | Списан / снят с учёта |
| `other` | storekeeper | Иная причина (`cancel_reason_text` обязателен) |

**UI CMMS:** при `cancelled_by = storekeeper` показать баннер с `cancel_reason` / `cancel_reason_text`; предложить создать новую заявку (другой склад или состав).

## Use-cases контура Б

### ISS-UC-1 — Создать заявку на получение инструмента

| | |
|---|---|
| Инициатор | Диспетчер CMMS |
| Вызов | ISS-API-3 (склады) → ISS-API-4 (каталог) → ISS-API-1 (создание) |
| Поведение CMMS | Форма: один склад за заявку; при необходимости другого склада — повторная заявка |

### ISS-UC-2 — Резерв строк кладовщиком

| | |
|---|---|
| Инициатор | Кладовщик TMS |
| Действие | Подбор конкретного серийного экземпляра под каждую строку → `line_status = reserved` |
| Альтернатива | Если выполнить заявку невозможно (нет остатка, брак и т.п.) — **ISS-UC-6b** (отмена всей заявки), а не «вечный» `new` / `partially_reserved` |
| Видимость CMMS | Через опрос ISS-API-5 |

### ISS-UC-3 — Выдача комплекта

| | |
|---|---|
| Предусловие | Производный `status = ready_for_issue` |
| Действие | Все строки → `issued` разом (частичной выдачи не бывает) |
| Видимость CMMS | Бейдж «Выдано» после следующего опроса |

### ISS-UC-4 — Отслеживание статуса диспетчером

| | |
|---|---|
| Инициатор | CMMS |
| Механизм | Опрос ISS-API-5 при открытии карточки наряда + периодически для активных заявок |
| Отмена кладовщиком | При `cancelled` + `cancelled_by = storekeeper` — показать причину; не оставлять заявку в «зависшем» `new` / `partially_reserved` |

### ISS-UC-5 — Возврат на склад

| | |
|---|---|
| Действие | Кладовщик TMS оформляет возврат всех строк разом → `returned` |
| Видимость CMMS | Через ISS-API-5, CMMS ничего не пишет в TMS |

### ISS-UC-6a — Отмена заявки диспетчером (CMMS)

| | |
|---|---|
| Инициатор | Диспетчер CMMS (отмена наряда или отдельной заявки) |
| Вызов | ISS-API-2 (batch) |
| Ограничение | Только пока ни одна строка не выдана (`status` ∈ `{new, partially_reserved, ready_for_issue}`) |
| Видимость TMS | `cancelled_by = dispatcher`, `cancel_reason` из тела запроса |

### ISS-UC-6b — Отмена заявки кладовщиком (TMS)

| | |
|---|---|
| Инициатор | Кладовщик TMS |
| Сценарий | Открыл заявку, не может зарезервировать/выдать комплект (нет в наличии, брак, списание) |
| Действие | Отменить **всю заявку** в UI TMS: выбрать `cancel_reason`, опционально `cancel_reason_text` |
| Ограничение | Те же производные статусы, что в ISS-UC-6a — до выдачи |
| Видимость CMMS | Следующий опрос ISS-API-5: `status = cancelled`, `cancelled_by = storekeeper`, причина; диспетчер может оформить новую заявку на другой склад |

> **Почему отмена целиком:** выдача только комплектом — до `ready_for_issue` каждая строка должна стать `reserved`. Если одна позиция недоступна, частичный резерв не приводит к выдаче; зависание в `partially_reserved` без обратной связи в CMMS недопустимо.

# За рамками MVP

Следующие возможности осознанно исключены из объёма и могут быть добавлены
отдельным этапом при необходимости:

- **Push-уведомление о выдаче/возврате инструмента** (TMS → CMMS webhook после `issued`/`returned`) — в MVP статус узнаётся только через опрос ISS-API-5.
- **Webhook при создании отчёта о работе** (контур А, учёт наработки/моточасов на инструменте) — заменяется опросом REP-API-3 при необходимости.
- **Supabase Realtime** между проектами (для обоих контуров) — требует отдельной проработки RLS-политик для внешнего сервисного доступа к репликации таблиц; в MVP не используется.
- **Условные запросы через `ETag`/`If-None-Match`** — оптимизация трафика опроса, не влияет на корректность работы MVP.
- **Ротация секретов интеграции** — секреты статичны на период MVP, механизм ротации согласуется отдельно.

# Глоссарий

| Термин | Значение |
|---|---|
| **CMMS** | BAAZ CMMS — АС ТОиР, заявки и отчёты по ремонту |
| **TMS** | BAAZ Tool Management System — учёт инструмента |
| **Контур А** | Ремонт/поверка инструмента, инициатор — TMS |
| **Контур Б** | Получение инструмента на наряд, инициатор — CMMS |
| **tool_id** (контур А) | UUID **конкретного физического экземпляра** инструмента в TMS |
| **catalog_item_id** (контур Б) | UUID **типа/каталожной позиции** в номенклатуре склада TMS — не экземпляр |
| **client_reference_id** | UUID, генерируемый вызывающей стороной для идемпотентности создания |
| **line_client_id** | Стабильный UUID строки заявки на выдачу, генерируется CMMS до первой отправки |
| **Снимок** | Копия данных на момент создания заявки, не обновляется при изменении источника |
| **Производный `status`** | Статус заявки на выдачу, вычисляемый TMS из `lines[].line_status` и `cancelled_at`, не хранится отдельной колонкой |
| **cancelled_by** | `dispatcher` (ISS-API-2) или `storekeeper` (UI TMS) |
| **cancel_reason** | Код причины отмены; набор значений зависит от `cancelled_by` |
| **schema_version** | Номер версии контракта в теле запроса/ответа |
